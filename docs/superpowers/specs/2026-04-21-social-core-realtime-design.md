# 校园社交核心能力与实时消息设计

## 背景

当前项目已经具备 uni-app 前端与 NestJS 后端的基础模块，上传、消息、评论、关注、点赞、收藏相关代码也已有部分雏形，但仍存在以下问题：

- 前端多个页面仍使用本地假数据或本地状态模拟。
- 后端点赞、收藏、关注、评论计数缺少幂等与一致性保障。
- 搜索功能尚未形成统一接口与真实结果页。
- 图片上传已有服务端雏形，但尚未接入发布流程。
- 消息功能只有 REST 基础接口，缺少实时推送层。

本次设计目标是在不推翻现有架构的前提下，先形成一个稳定的 REST 功能闭环，再在其上补充 WebSocket 实时推送。

## 目标

- 支持发布动态时上传图片并提交可访问的图片 URL。
- 支持搜索动态、用户、资源，并在搜索页展示真实结果。
- 支持动态详情页真实加载评论、发布评论、回复评论、点赞评论。
- 支持用户主页和动态详情页的关注/取消关注，并修复粉丝与关注列表逻辑。
- 支持动态点赞/收藏状态同步，避免重复计数和本地假状态。
- 在消息 REST 能力稳定后，增加 WebSocket 实时消息推送与未读同步。

## 非目标

- 不做新的推荐算法、热门排序、全文检索引擎接入。
- 不在本轮引入 Redis、消息队列或横向扩展方案。
- 不重构整个前端视觉或全站交互模式。
- 不新增复杂的审核、举报、黑名单系统。

## 方案概览

整体分为两个阶段：

### 第一阶段：REST 功能闭环

围绕上传、搜索、评论、关注、点赞、收藏、消息基础 REST 完成真实接口接入和前端落地。该阶段完成后，用户即使没有实时连接，也可以通过正常请求完成全部核心操作，并且所有页面状态来源统一由后端提供。

### 第二阶段：WebSocket 实时消息推送

在现有消息 REST 发送、查询、已读逻辑保持不变的前提下，增加一条独立的实时通道：

- 用户进入聊天页时建立 WebSocket 连接。
- 发送消息时仍落库，由服务端在落库成功后推送给接收方。
- 聊天列表和未读计数在收到推送后局部刷新。
- 连接断开后客户端自动重连。

该顺序可以把数据正确性和交互体验分层处理，避免“实时层掩盖基础数据不可靠”的问题。

## 现有代码约束

当前代码中已有如下可复用基础：

- `Server/src/modules/upload` 已存在上传控制器与模块骨架。
- `User` 实体已定义 `following`、`likedPosts`、`collectedPosts`、`collectedResources` 关系。
- `Post`、`Comment`、`Message` 已有基础实体与控制器。
- `Clint/src/pages/home/home.vue`、`post-detail.vue`、`search.vue`、`user-profile.vue`、`chat.vue` 已有页面骨架。

但也有已知缺陷：

- 点赞/收藏目前直接改计数字段，未基于关系表去重。
- `getFollowers` / `getFollowing` 逻辑存在语义错误。
- `post-detail`、`search`、`user-profile`、`chat` 仍大量使用本地模拟数据。
- 评论与帖子评论数未联动更新。
- 详情与列表未返回 `isLiked`、`isCollected`、`isFollowing`。

设计会在沿用现有模块划分的前提下修复这些问题，而不是大规模重构。

## 后端设计

### 1. 图片上传

沿用已有 `upload` 模块，补足以下能力：

- 启动时确保上传目录存在。
- 上传接口返回标准化图片 URL。
- 文件类型仅允许常见图片 MIME。
- 限制单图大小与单次上传数量。

前端发布动态时先调用上传接口，再将返回的 URL 数组提交给创建动态接口。

### 2. 点赞与收藏状态同步

点赞和收藏不再只依赖 `likesCount` / `collectionsCount` 的自增自减，而是以关系表为真实来源：

- 点赞动态：检查 `user_liked_posts` 是否已有关系。
- 收藏动态：检查 `user_collected_posts` 是否已有关系。
- 收藏资源：检查 `user_collected_resources` 是否已有关系。

行为规则：

- 已点赞再次点赞，返回成功但不重复加计数。
- 未点赞时取消点赞，不允许减为负数。
- 收藏与取消收藏遵循相同规则。

接口返回中补充：

- `isLiked`
- `isCollected`

列表页和详情页都应返回这两个字段，以便前端不再维护脱离服务端的假状态。

### 3. 关注系统

关注逻辑继续基于 `User.following` 多对多关系，但修复当前实现：

- 关注自己时返回错误。
- 重复关注不重复增加粉丝数和关注数。
- 取消未关注用户不产生负数。
- `getFollowers` 返回“关注我的用户列表”。
- `getFollowing` 返回“我关注的用户列表”。

用户详情接口补充：

- `isFollowing`

这样动态详情页、用户主页均可直接消费。

### 4. 评论系统

评论继续使用现有 `Comment` 实体，支持：

- 帖子一级评论
- 对某条评论进行回复
- 评论点赞/取消点赞

约束与同步规则：

- 创建评论时，帖子 `commentsCount` 增加。
- 删除评论时，帖子 `commentsCount` 减少，但不低于 0。
- 拉取评论列表时返回作者、回复对象、子回复信息。
- 评论项返回 `isLiked`，用于详情页渲染真实状态。

前端初版以“详情页展示一级评论并支持回复”为主，不在本轮扩展复杂的折叠树和多级分页。

### 5. 搜索系统

新增统一搜索模块，例如 `GET /search`：

- `q`：关键词
- `type`：`all | posts | users | resources`
- `page`
- `limit`

搜索行为：

- 动态：按内容、标签、位置模糊匹配。
- 用户：按昵称、学校、学院、专业模糊匹配。
- 资源：按标题、描述、标签模糊匹配。

响应结构：

- 当 `type=all` 时返回 `{ posts, users, resources }`
- 当指定类型时返回该类型分页结果

初版采用 TypeORM `LIKE/ILIKE` 查询即可，优先满足可用性。

### 6. 消息 REST 与实时推送

REST 基础能力保持：

- 获取会话列表
- 获取与某用户的聊天记录
- 发送消息
- 标记已读
- 获取未读数

在此基础上新增 WebSocket 网关：

- 握手阶段基于 JWT 鉴权。
- 每个在线用户加入 `user:<id>` 房间。
- 新消息落库成功后，向接收者房间推送 `message:new`。
- 接收方当前正在该会话中时，前端可立即插入消息并调用已读接口。
- 连接断开后从房间中清理。

初版不引入 Redis 适配器，默认单实例部署。

## 前端设计

### 1. 发布页

`publish.vue` 调整为真实发布流程：

- 选择图片后先显示本地预览。
- 点击发布时依次上传未上传图片。
- 上传成功后调用创建动态接口提交内容、分类、图片 URL。
- 发布成功后返回上一页并触发首页刷新。

### 2. 首页与动态详情

首页与详情页改为统一消费真实字段：

- `isLiked`
- `isCollected`
- `author.isFollowing` 或等价关注状态

交互规则：

- 用户点击点赞/收藏后先调用接口，再更新本地视图。
- 接口成功时更新本地计数和状态。
- 接口失败时保持原状态并提示错误。

详情页增加真实评论区：

- 进入页面时加载帖子详情与评论列表。
- 发送评论后插入新评论并同步评论数。
- 点赞评论时以接口返回状态刷新单条评论。

### 3. 搜索页

搜索页从静态示例切为真实查询：

- 输入关键词后防抖查询。
- 默认展示综合结果。
- 支持切换动态、用户、资源三个 tab。
- 保留本地搜索历史，热门搜索仍可先用静态配置。

### 4. 用户主页

`user-profile.vue` 改为真实加载：

- 用户基本资料
- 关注状态
- 用户动态列表

点击关注后使用真实接口更新按钮与粉丝数。

### 5. 聊天页

聊天页分两步接入：

第一步：

- 真实加载聊天记录
- 真实发送消息
- 进入会话时标记已读

第二步：

- 建立 WebSocket 连接
- 收到 `message:new` 事件后插入消息
- 新消息到达时滚动到底部并刷新未读状态

## 数据流设计

### 点赞/收藏

1. 前端请求点赞/收藏接口。
2. 后端检查关系表中是否已存在关系。
3. 后端决定创建关系、删除关系或忽略重复动作。
4. 后端返回最新状态与计数。
5. 前端使用返回值刷新本地视图，而不是自行猜测。

### 评论

1. 前端提交评论内容与目标帖子/父评论信息。
2. 后端写入评论。
3. 后端同步更新帖子评论数。
4. 后端返回完整评论对象。
5. 前端插入列表并更新评论总数。

### 实时消息

1. 客户端连接 WebSocket，并携带 JWT。
2. 服务端鉴权成功后将连接绑定到当前用户房间。
3. 用户发送消息仍走 REST，服务端完成落库。
4. 服务端向接收方房间广播新消息事件。
5. 接收方页面收到消息后更新聊天窗口和未读数量。

## 错误处理

### 后端

- 非法文件类型或超限文件返回明确错误信息。
- 重复关注、重复点赞、重复收藏视为幂等成功，不抛服务端异常。
- 关注自己、访问不存在资源、越权修改内容返回明确业务错误。
- WebSocket 鉴权失败立即拒绝连接。

### 前端

- 上传失败时保留已选图片，提示用户重试。
- 搜索失败时显示空态或失败提示，不清空已有输入。
- 点赞/收藏/关注失败时不进行乐观反转。
- WebSocket 断开后自动重连，并在 UI 上保持静默恢复。

## 实施顺序

建议按以下顺序落地：

1. 修复后端点赞、收藏、关注、评论数的一致性。
2. 为帖子、资源、用户详情和列表补充状态字段。
3. 接入前端首页、详情页、用户主页的真实状态。
4. 接入图片上传与发布闭环。
5. 增加搜索接口并接入搜索页。
6. 接入聊天页 REST 能力。
7. 增加 WebSocket 推送与客户端重连。

## 风险与取舍

- 当前项目没有专门的迁移文件，若数据库关系表与实体不一致，联调时可能需要补初始化或手工修正数据库。
- 若小程序端 WebSocket 与 H5 表现存在差异，需要在连接与重连逻辑上做平台兼容处理。
- 搜索初版基于数据库模糊查询，在数据量大时性能一般，但对当前阶段足够。
- 为控制范围，本轮只做消息实时推送，不把点赞、评论通知也改成实时。

## 完成标准

当满足以下条件时，本轮工作视为完成：

- 用户可以上传图片并发布带图动态。
- 用户可以真实搜索动态、用户和资源。
- 用户可以查看、发布、回复、点赞评论。
- 用户可以真实关注/取消关注，粉丝和关注列表语义正确。
- 用户在列表与详情中看到一致的点赞/收藏状态。
- 聊天页可以通过 REST 正常收发消息，并在在线情况下收到 WebSocket 实时推送。
